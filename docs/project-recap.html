Decision Log

Cookie-based sessions (not JWT)
Server-side sessions allow explicit revocation, TTL enforcement, and fit the stateless API model. JWTs would require blacklist or short TTLs for logout.
Invite state via LobbyMember, not separate model
Pre-registration invites stored as LobbyMember with user_id=None, target_email, status=invited. Keeps lobby state visible before account creation without separate projection layer.
Lobby-scoped auth composes get_current_user + LobbyMember lookup
Every lobby endpoint verifies caller is a member via lobby_id + user_id lookup, returns 403 when not a member. Enforces strict per-lobby isolation.
Email normalization to lowercase + trimmed
Enforces global uniqueness via DB index. GM accounts and player accounts share email namespace — same email can't be both GM and player.
SQLite dev, PostgreSQL prod
Fast iteration locally, production-ready for deployment. No Alembic migrations yet — tables created via Base.metadata.create_all().

Mental Model Essentials

  • GM-only lobby creation: Only users with account_type = GM can create lobbies. Players cannot create lobbies.
  • One DM per lobby: The creator is automatically the DM (is_dm=True, status=active). Invariant enforced in code.
  • Email is globally unique: Same email can't be both GM and player. Need separate accounts to participate in different roles.
  • Invites reuse LobbyMember: Pending invites are LobbyMember rows with user_id=None, target_email set, status=invited.
  • Invite tokens are single-use: Token expires in 7 days, marked used_at on acceptance. Not reusable.
  • Session auth via cookie: get_current_user reads session_id cookie, validates session exists and not expired/revoked.
  • Lobby isolation: All lobby endpoints check membership (lobby_id + user_id). No cross-lobby access possible.
